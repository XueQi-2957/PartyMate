## 3. PartyMate 项目介绍

### 3 分钟项目介绍

PartyMate 是我做的一个本地党务智能 Agent 工作台，目标用户是高校基层党务负责人。这个场景来自我的真实经历：发展党员材料很多，格式要求严格；会议记录要整理成规范纪要；成员处在不同发展阶段，缺什么材料、什么时候该提醒，人工维护很容易漏。

所以我没有把它做成一个普通聊天机器人，而是做成一个有业务状态的 Agent 工作台。它覆盖几条流程：材料检查、材料包导入、会议纪要整理、会议待办写入提醒、成员台账、成员记忆、知识库问答，以及 Agent 执行记录。

技术架构上，后端是 Python + Starlette + Uvicorn，前端是原生 HTML/CSS/JS 的本地 SPA，数据库是 SQLite。模型侧支持本地 Ollama，也兼容 OpenAI-compatible API。Agent runtime 是我自己基于 httpx 和 function calling 写的。每次用户发起请求后，系统组装 system prompt、会话上下文、成员上下文和工具 schema；模型如果决定调用工具，就返回 tool calls；后端执行白名单工具，把结果作为 observation 回填给模型，最多循环 10 轮，最后输出答案。

我觉得这个项目比较有价值的地方有三个。第一是 Agent 可观测性。每次 Agent run 都会记录用户输入、模型、工具调用链、参数、结果摘要、耗时和状态，方便排查模型为什么这么回答。第二是成员级记忆。每个成员有独立 memory namespace，Agent 只加载当前成员的档案和记忆，避免不同成员上下文串扰。第三是任务闭环。比如会议整理后可以抽取待办并写入提醒；材料包上传后可以解压、识别材料类型、OCR、人工复核，再做缺件、重复、阶段冲突和身份一致性检查。

我后续想继续优化的是，把复杂流程进一步抽象成更明确的状态机，引入更系统的评测集，增强工具权限控制和 API schema 校验。如果项目要产品化，我会考虑迁移到 FastAPI 或增加 Pydantic 请求模型，但当前 Starlette 对本地单用户 Agent 工作台来说比较轻量和可控。

### 30 秒版本

PartyMate 是一个面向高校党务场景的本地 Agent 工作台，用 Python + Starlette + SQLite + Ollama 实现。它围绕材料审核、党务知识库问答、会议整理、待办提醒和成员台账做多工具协同。核心不是聊天，而是让 Agent 能检索规则、调用工具、读取成员上下文、生成结果并回写提醒或记录。同时我实现了 Agent trace，记录每次运行的工具调用链和耗时，用来解决 Agent 黑盒问题。

## 4. PartyMate 高频追问与回答

### Q1：为什么做 PartyMate？

回答：

这个项目来自我自己的党务工作经历。基层党务里有很多重复但容易出错的工作，比如发展党员材料格式审核、思想汇报缺件、会议纪要整理、会议后待办跟踪、成员阶段推进等。通用 AI 可以帮忙写文案，但它不知道具体成员处在哪个阶段，也不会自动回写台账或提醒。所以我想做一个本地优先、有状态、可追溯的 Agent 工作台，把模型能力接到具体工具和业务流程里。

可追问补充：

这个项目的价值不是“生成一段党务文本”，而是把多个小任务串成闭环：读文件、检索规则、检查材料、生成问题清单、需要人工复核的地方标出来，再把结果沉淀到成员档案或提醒系统里。

### Q2：为什么说它是 Agent，不是普通 AI 应用？

回答：

我理解 Agent 至少要有几个特征：能理解任务、能选择工具、能根据工具结果继续推理、能维护上下文状态，并且最好有执行过程记录。PartyMate 里模型不是一次性回答，它会拿到一组 function schema，比如材料检查、会议整理、知识库检索、成员查询、提醒查询等。模型决定调用哪个工具后，后端执行工具，把结果作为 observation 回填给模型。如果信息还不够，模型可以继续调用下一步工具，直到形成最终回答。

区别在于：普通 AI 应用更像“prompt in, answer out”；PartyMate 更像“任务 in, 多工具执行过程 out, 结果可追踪”。

### Q3：Agent Runtime 是怎么实现的？

回答：

核心在 `partymate/agent.py`。流程是：

1. 从环境变量读取模型配置，支持 Ollama 或 OpenAI-compatible API。
2. 读取会话记忆和成员上下文，组装 messages。
3. 从 `AgentToolRegistry` 获取工具 schema。
4. 用 httpx 调 `chat/completions`，开启 stream。
5. 如果模型返回 tool_calls，就解析函数名和参数。
6. 后端执行对应白名单工具，把结果作为 tool message 回填。
7. 最多循环 10 轮，没有工具调用时返回最终回答。
8. 同时把工具调用和结果写入 Agent trace。

一句话概括：这是一个轻量版 ReAct + function calling loop。

### Q4：为什么不用 LangChain / LangGraph？

回答：

我了解 LangChain 和 LangGraph 的生态，但 PartyMate 当前核心 runtime 是自己写的。原因有两个：第一，项目初期我想把 tool calling、上下文组织、工具结果回填、trace 这些底层机制真正理解清楚；第二，PartyMate 是本地单用户工作台，流程还没有复杂到必须用 LangGraph 编排。

如果后续要做更复杂的多节点流程，比如材料包审核拆成“解压、解析、OCR、人工复核、规则核查、报告生成、回写台账”这些明确节点，我会考虑把复杂流程迁移到 LangGraph。现在手写 runtime 的好处是轻量、可控、便于调试。

### Q5：你在 PartyMate 里用了哪些工具？

回答：

主要分几类：

第一类是业务工具，比如 `check_doc` 检查发展党员材料，`meeting_summary` 整理会议纪要，`content_gen` 生成三会一课方案。

第二类是数据查询工具，比如查询成员、查询材料、查询提醒、获取 dashboard summary。

第三类是知识工具，比如 `search_knowledge`，通过 MCP 调用 RAG 检索党务知识库。

第四类是写入类工具，比如创建提醒、添加成员记忆。对于写入类工具，我的设计倾向是要经过确认，不让模型直接静默写入敏感数据。

第五类是辅助工具，比如获取当前时间、天气查询、受控的 Python 执行请求。

### Q6：Tool Calling 的 schema 怎么设计？

回答：

每个工具都暴露成 OpenAI-compatible function schema，包含 name、description、parameters、required 字段。设计时我会尽量让参数结构简单、明确，比如材料检查只需要 `text`，会议整理只需要 `text`，成员材料查询需要 `member_id`。description 要写清楚什么时候调用，因为模型主要靠工具描述做路由。

我也会区分读操作和写操作。读操作可以直接执行；写操作更谨慎，应该返回待确认状态或由前端弹确认卡片，避免模型误操作。

### Q7：工具调用失败怎么办？

回答：

目前工具执行外层有 try/except，失败时会返回错误信息给模型，模型可以基于错误继续回答或提示用户。API 调用模型时也会检查 HTTP 状态码，如果模型服务失败，会通过 SSE error 事件返回给前端。

如果要做得更生产化，我会补三类机制：工具级超时控制、可重试错误和不可重试错误分类、以及针对关键工具的结构化错误码。这样模型看到的不只是“出错了”，而是能知道是参数缺失、文件不存在、权限不足，还是模型服务不可用。

### Q8：RAG 链路怎么做的？

回答：

PartyMate 的 RAG 主要服务党务规则检索。知识库来自本地党务规程和工作手册。链路大概是：

1. 先从 DOCX 或规则文档中提取文本。
2. 做清洗和分块，优先按法规章节、流程节点和语义边界分块。
3. 建 ChromaDB 向量索引，嵌入模型使用 Ollama 的 bge-m3。
4. 同时建 BM25 稀疏检索索引。
5. 查询时做 dense + sparse 的混合召回。
6. 对候选结果做 rerank，当前代码里支持 cross-encoder 或 LLM rerank。
7. 最后把引用片段注入 Agent 上下文或返回给普通 API。

这里做混合检索是因为党务材料里有很多固定术语，比如“发展对象”“预备党员”“支部大会”“政审”，只用向量检索可能召回语义相近但术语不精确的内容；BM25 能补关键词精确匹配。

### Q9：为什么用混合检索，不只用向量库？

回答：

党务知识库是强术语、强流程的文档。只用向量检索，模型可能把语义相近但流程节点不同的段落召回来；只用 BM25，又容易漏掉表达不同但含义相近的内容。所以我采用混合检索：向量检索负责语义召回，BM25 负责关键词精确匹配，再通过 rerank 提高排序质量。

举例说，用户问“积极分子满多久能列为发展对象”，向量检索能找到培养考察相关段落，BM25 能抓住“积极分子”“发展对象”“一年以上”这些关键字，两者融合更稳。

### Q10：RAG 怎么保证回答可追溯？

回答：

我要求工具返回带 id、标题和内容片段的检索结果。Agent 的 system prompt 里也要求，使用知识库或联网搜索结果时，回答要引用工具返回的 id，不能自己编造来源。普通 API 也会把 `format_citations` 的结果一起返回给前端，让用户看到依据片段。

不过我不会说它已经完全解决幻觉。更准确的说法是：RAG 能降低无依据回答的概率，trace 和 citation 能让用户回查依据，但仍然需要评测和人工复核，尤其是党务这种规范性场景。

### Q11：RAG 分块怎么做？

回答：

当前代码里有两套逻辑。优先尝试 LLM 语义分块，让模型按章节、条款、流程步骤拆成 150 到 1500 字左右的知识块。如果 LLM 不可用，就回退到规则分块，按 Markdown 标题和段落边界切分。这样做是因为党务规程天然有章节和步骤，不能随便按固定 token 数切，否则可能把一个流程节点切断。

如果面试官追问不足：

目前分块还可以继续优化，比如增加 chunk overlap 的评测、对表格和模板单独处理、构造问题集来评估不同 chunk size 对召回率的影响。

### Q12：你怎么做成员级记忆？

回答：

成员级记忆是为了避免不同发展对象之间的上下文串扰。每个成员有独立 memory namespace，记忆类型包括 summary、risk、instruction、correction、note。记忆内容存 SQLite，不和普通聊天历史混在一起。

当用户在某个成员页面发起 Agent 对话时，后端会调用 `MemberMemoryService.build_agent_context`，把成员姓名、阶段、专业、学号、备注以及重要记忆组装成一段 system context 注入模型。模型只围绕当前成员给建议。

### Q13：成员记忆和聊天历史有什么区别？

回答：

聊天历史是会话级的，主要服务当前对话连续性；成员记忆是业务级的，服务长期任务。比如用户这次说“张三的思想汇报时政结合不够”，这条信息如果只是聊天历史，下次换会话可能丢失；如果沉淀成成员记忆，下次进入张三详情页时仍然能加载。

我会把它们分开，是因为聊天历史可能很长、噪声很多，而成员记忆应该是筛选后的高价值事实或指令。

### Q14：会话记忆怎么做？

回答：

会话记忆在 `chat_memory.py` 里。它有三层：

1. Sliding window，保留最近几轮原始对话。
2. Incremental summary，对被挤出窗口的历史做摘要。
3. Vector recall，把历史对话片段入 ChromaDB，根据当前问题召回相关片段。

这样做是为了兼顾短期精确上下文和长期历史召回。当前实现还是原型级，后续我会加强摘要持久化和召回质量评测。

### Q15：Agent Trace 是怎么设计的？

回答：

每次 Agent run 开始时生成 run_id，写入 `agent_runs`，状态是 running。每次工具执行后写一条 tool call 记录，包括 tool_name、arguments_json、result_summary、duration_ms、call_order。最终回答生成后更新 run 状态、总耗时、结果摘要和工具调用列表。

这个设计主要解决两个问题：第一，排查问题时知道模型到底调用了什么工具；第二，用户可以回看执行链路，尤其是涉及材料审核和党务建议时，不能完全黑盒。

### Q16：会议闭环怎么做？

回答：

会议模块分两步。第一步用 `parse_meeting_notes` 从原始会议记录中抽取日期、主持人、议题、决议、待办等结构化字段。第二步用 `MeetingWorkflowService.parse_and_write_actions` 从 action_items 或文本规则中提取待办事项，写入 meeting_actions。如果绑定了成员，还会写入 reminders。待办有 pending、confirmed、done 三种状态。

我想表达的是：工具输出不只停留在文本，而是进入后续任务系统。

### Q17：材料包导入怎么做？

回答：

材料包导入在 `MaterialImportService`。用户上传 ZIP 后，系统会：

1. 创建 import batch。
2. 把原始 ZIP 保存到受控目录。
3. 解压每个支持格式的文件。
4. 做 zip slip 防护，避免 `../evil.txt` 这类路径穿越。
5. 调用 `parse_file` 解析 PDF、Word 或图片。
6. 根据文件名、别名和正文识别材料类型。
7. 保存全文和材料记录。
8. 如果是图片 OCR 且低置信度，就创建 OCR review task。

后续整套核查会基于这个 batch 的材料记录执行。

### Q18：材料一致性核查检查哪些东西？

回答：

主要检查五类：

1. 缺件：根据成员当前发展阶段，检查当前阶段及之前阶段的必备材料。
2. 重复：同一类型材料是否重复导入，思想汇报这类周期性材料例外。
3. 阶段冲突：比如成员还在申请人阶段，却出现预备党员材料。
4. 身份冲突：材料正文中的姓名、专业和成员档案是否一致。
5. 待复核：OCR 低置信度、未知类型、解析失败等文件进入人工复核。

### Q19：OCR 人工复核闭环是什么？

回答：

图片或扫描件 OCR 可能识别错，尤其是党务材料里有手写签名、扫描件、表格。我的设计不是宣称全自动，而是检测低置信度片段或文本过短的情况，创建 OCR task。用户进入复核页面，查看原始 OCR 文本和置信度摘要，修正确认后，系统把 confirmed text 写入库。后续材料核查优先使用确认文本。

这体现的是人机协同：机器做初筛，关键字段让人确认。

### Q20：为什么用 Starlette？

回答：

PartyMate 是本地单用户 Agent 工作台，不是传统多用户后台系统。它主要需要轻量 API、静态文件挂载、异步处理、SSE 流式响应和 lifespan 初始化资源。Starlette 是轻量 ASGI 框架，适合做这种 Web 胶水层。

Django 很适合 CMS、后台管理、用户权限、ORM 强绑定的业务平台，但 PartyMate 当前不需要 Django Admin、复杂认证和模板系统。FastAPI 也很适合，尤其是请求模型校验和自动文档；后续如果团队协作或产品化，我会考虑迁移到 FastAPI。当前选 Starlette 是为了轻量、可控、便于接 Agent streaming 和 MCP session。

### Q21：Starlette 和 Django 区别？

回答：

Django 是全家桶，高层框架，内置 ORM、Admin、认证、模板、表单、安全中间件，适合传统 Web 平台。Starlette 是轻量 ASGI toolkit，提供路由、请求响应、静态文件、WebSocket、StreamingResponse、lifespan 等基础能力，不绑定 ORM 和项目结构。

PartyMate 的复杂度主要在 Agent runtime 和 service 层，不在用户权限或后台管理，所以 Starlette 更合适。

### Q22：为什么不用 FastAPI？

回答：

FastAPI 也是很好的选择，而且它底层就是 Starlette。PartyMate 起步时我更关注 Agent 流程和业务闭环，路由层比较薄，所以直接用了 Starlette。FastAPI 的优势是 Pydantic schema、自动 OpenAPI 文档和参数校验。如果后续接口规模扩大，或者多人协作，我会优先迁移到 FastAPI，而不是直接上 Django。

### Q23：为什么用 SQLite？

回答：

因为 PartyMate 定位是本地优先、单用户、单支部工作台。SQLite 不需要额外部署数据库服务，数据就在本机文件里，适合本地工具和隐私敏感场景。它足够支撑成员、材料、提醒、记忆、OCR task、Agent run 这些结构化数据。

如果以后变成多用户平台，我会迁移到 PostgreSQL 或 MySQL，并重新设计用户权限和组织隔离。

### Q24：为什么本地优先？

回答：

党务材料包含姓名、学号、专业、政治面貌、申请材料等敏感信息，不适合默认上传到外部服务。本地优先可以让基础功能和本地模型在用户电脑上运行，减少隐私风险。外部 API 可以作为可选增强，但不是强依赖。

### Q25：你怎么处理安全问题？

回答：

我主要从三个层面处理：

第一是工具白名单，模型只能调用我暴露的工具，不能随意执行系统命令或访问任意文件。

第二是文件路径边界，比如 ZIP 解压时做 zip slip 防护，下载接口也禁止 `..` 路径。

第三是写操作确认。提醒创建、成员记忆写入、Python 代码执行这类敏感操作不应该让模型直接静默执行，要经过前端确认或人工审批。

如果要继续强化，我会加更严格的权限模型、路径 sandbox、工具风险等级和审计日志。

### Q26：`run_python_code` 会不会危险？

回答：

是的，这是高危能力，所以我在工具描述里明确标记为高危，并设计成需要用户批准后才实际执行。更生产级的做法应该是放到沙箱环境里，限制文件系统、网络、执行时间和可用模块，或者干脆只提供受控的数据处理工具，而不是任意代码执行。

面试时可以主动承认：这个点是安全风险，也是后续重点优化项。

### Q27：你怎么测试 PartyMate？

回答：

我写了针对核心 service 和 API 的单元测试。比如材料包导入测试里会 mock `parse_file`，验证 ZIP 路径穿越防护、 unsupported file 跳过、材料类型识别、全文持久化、低置信度 OCR 创建复核任务。Web API 测试用 Starlette TestClient 验证上传材料包、材料核查、OCR 确认、成员记忆 CRUD 和 dashboard 数据结构。

当前测试主要覆盖业务服务和 API 形状。后续我会补 Agent 层的 mock model 测试、RAG 评测集和端到端 UI 流程测试。

### Q28：Agent 怎么做流式输出？

回答：

后端 `api_chat_stream` 返回 `StreamingResponse`，media type 是 `text/event-stream`。Agent runtime 在执行过程中 yield 不同事件，比如 thinking、tool_call、tool_result、content、done、error。前端可以根据事件类型渲染模型思考、工具调用状态和最终文本。

这样用户不用等整轮任务结束，能看到 Agent 正在查知识库、调用工具或整理结果。

### Q29：你对 ReAct 的理解是什么？

回答：

ReAct 是 Reasoning + Acting。模型不是一次性给答案，而是在推理过程中决定采取行动，比如调用工具；拿到 observation 后再继续推理。它适合需要外部信息或多步执行的任务。

在 PartyMate 里，用户问一个材料问题，模型可能先调用知识库检索党务规程，再调用材料检查工具，最后综合工具结果回答。这个过程比单 prompt 更可控，也更容易追踪。

### Q30：PartyMate 最大的技术难点是什么？

回答：

我觉得难点不是某个单点算法，而是把 Agent 放进真实业务流程里。具体来说有三个：

第一，工具边界怎么设计。工具太粗，模型不可控；工具太细，调用链复杂、成本高。

第二，上下文怎么组织。要区分系统规则、会话历史、成员档案、成员记忆和 RAG 片段，否则上下文会乱。

第三，结果怎么闭环。Agent 不能只生成文本，还要把待办、核查结果、记忆沉淀到结构化系统里，同时保留人工复核入口。

### Q31：PartyMate 当前有什么不足？

回答：

主要有几类不足：

1. Agent workflow 还偏轻量 ReAct loop，没有用状态机明确每个任务节点。
2. RAG 还缺系统评测，比如命中率、citation accuracy、不同 top_k 和 chunk size 的对比。
3. 写操作确认机制还可以做得更完整。
4. 前端是原生 JS，适合原型，但如果产品化，可以升级 Vue/React。
5. 当前是单用户本地应用，还没有多用户权限、组织隔离和服务端部署能力。

我面试时会把它定义为“能展示 Agent 工程思路的本地工作台原型”，而不是成熟 SaaS 平台。

## 5. MiniCode 项目追问与回答

### Q1：MiniCode 是什么？

回答：

MiniCode 是我参考 Claude Code 交互范式做的 Coding Agent 原型，目标是理解 AI Coding Agent 怎么在真实代码环境中持续执行任务。它不是简单代码生成器，而是围绕 Query Loop + Tool Use 实现任务闭环：模型理解用户任务，选择工具读取文件、搜索代码、修改文件或运行检查，把工具结果回填，再继续下一步。

### Q2：为什么要做 MiniCode？

回答：

我做 PartyMate 更偏业务 Agent，MiniCode 则更偏通用开发 Agent。Coding Agent 对工具调用、上下文管理、权限控制要求更高，因为它会读写真实文件、执行命令、跨多轮修复错误。我想通过 MiniCode 理解 Claude Code 这类产品背后的执行循环，而不是只会使用工具。

### Q3：Query Loop 是什么？

回答：

Query Loop 可以理解成 Agent 的主循环。每一轮包含：

1. 接收用户目标和当前上下文。
2. 调模型，让模型决定下一步是回答还是调用工具。
3. 如果调用工具，就解析工具名和参数。
4. 执行工具，把结果回填给模型。
5. 根据新 observation 进入下一轮。
6. 直到任务完成、失败或达到轮次上限。

Coding Agent 的关键是循环要稳定，不能工具调用一次就结束，也不能无限循环。

### Q4：Tool Use 在 Coding Agent 里有什么难点？

回答：

难点主要是工具风险和结果解释。

读文件、搜索代码风险较低；写文件、移动文件、运行命令风险较高；删除文件、git reset、执行网络命令风险更高。不同工具要有不同权限策略。

另一个难点是工具结果可能很长，比如搜索结果、测试日志、diff。如果不做裁剪和摘要，会占满上下文。所以 Coding Agent 需要把工具输出转成模型能继续推理的 observation。

### Q5：Skill 分层路由是什么？

回答：

我把能力分成三层：底层是原子工具，比如读文件、写文件、搜索、执行命令；中层是 Skill，比如调试流程、前端修改流程、文档生成流程；上层是 Skill 目录或元信息，用来检索候选 Skill。

随着 Skill 变多，不能每次把所有 Skill 全塞进 prompt，否则 token 成本高，模型也容易选错。所以需要先根据任务意图、标签、描述和示例召回候选 Skill，再精排选择。

### Q6：Skill 路由怎么降低 token 成本？

回答：

如果有几十个甚至上百个 Skill，把全部说明放进上下文会很贵，而且会制造噪声。更合理的是只放目录级元信息，先召回少量候选，比如 3 到 5 个，再把完整 Skill 内容加载进上下文。这样既降低 token，也提高模型选择工具的准确性。

### Q7：上下文压缩怎么做？

回答：

长任务里上下文会快速膨胀，尤其是代码搜索结果、错误日志、工具 observation。我的思路是把上下文分成几类：

1. 当前任务目标，必须保留。
2. 最近几轮对话和工具结果，原文保留。
3. 较早历史，压缩成摘要。
4. 项目约束和用户偏好，沉淀为记忆。
5. 可重新获取的信息，比如文件全文，不一定长期塞在上下文里，需要时再读。

压缩不是简单截断，而是保留决策、错误、已尝试方案和当前约束。

### Q8：Coding Agent 的记忆应该记什么？

回答：

我认为应该记四类：

1. 程序性经验，比如这个项目测试命令是什么、构建命令是什么。
2. 错误修复经验，比如某个依赖版本冲突怎么处理过。
3. 用户偏好，比如代码风格、是否喜欢先写测试。
4. 项目约束，比如不能改某些目录、不要引入新依赖。

不应该把所有聊天内容都记下来，否则记忆会变成垃圾堆。

### Q9：权限审查怎么做？

回答：

Coding Agent 的权限审查可以分几层：

第一，规则过滤，比如禁止访问工作区外路径、禁止危险命令、删除操作需要确认。

第二，工具自检，比如写文件前检查路径是否在 workspace 内，执行命令前判断是否含网络、删除、git reset 等风险。

第三，AI 风险分类，让模型辅助判断操作意图，但不能只依赖模型。

第四，人工确认，对高风险操作弹确认。

核心原则是：低风险读操作自动执行，高风险写操作和系统操作要可审计、可确认。

### Q10：多 Agent 协作你怎么理解？

回答：

多 Agent 不一定是越多越好。我理解它适合把不同职责拆开，比如一个负责代码阅读，一个负责实现，一个负责测试，一个负责审查。关键不是“起很多 Agent”，而是职责边界清楚、共享上下文可控、最终由 orchestrator 汇总决策。

如果任务很简单，单 Agent + 工具就够了。多 Agent 的成本是通信复杂、上下文重复和一致性问题。

### Q11：MiniCode 和 PartyMate 的区别？

回答：

PartyMate 是垂直业务 Agent，重点是党务材料、会议、成员台账这些业务流程闭环。MiniCode 是开发工具型 Agent，重点是文件读写、代码理解、工具执行、安全权限和长任务上下文。

一个解决“业务流程自动化”，一个解决“开发任务自动化”。但底层都有共性：tool calling、context management、memory、trace、安全控制。

### Q12：如果让你继续做 MiniCode，你会优先做什么？

回答：

我会优先做三件事：

1. 更严格的 sandbox 和权限模型，尤其是命令执行、文件写入和 git 操作。
2. 更稳定的上下文压缩和任务状态管理，让长任务不丢关键约束。
3. 做一套 benchmark，比如给定小型 repo 和 bugfix 任务，评估成功率、工具调用次数、错误恢复能力和最终 diff 质量。

## 6. Agent 技术基础高频题

### Q1：什么是 AI Agent？

回答：

我理解 AI Agent 是以大模型为决策核心，能够感知上下文、规划下一步、调用工具、观察结果并持续推进任务的系统。它和普通 LLM 应用的区别在于，Agent 不只是生成文本，而是能通过工具影响外部环境，比如查数据库、读文件、检索知识库、生成报告、写入待办。

### Q2：Agent 和 Workflow 有什么区别？

回答：

Workflow 是预定义流程，节点和顺序通常是固定的，比如先 OCR，再检查，再生成报告。Agent 更强调模型根据当前上下文动态决定下一步，比如先查知识库还是先读文件由模型判断。

实际工程里我觉得两者应该结合：确定性强的流程用 workflow，开放性强的决策点用 Agent。比如材料包处理的解压、OCR、入库适合 workflow；根据问题选择检索规则还是查询成员状态适合 Agent。

### Q3：什么是 Function Calling？

回答：

Function Calling 是让模型输出结构化工具调用，而不是自然语言说“我想查一下”。开发者提供工具 schema，包括工具名、描述和参数 JSON schema。模型根据用户问题选择工具并生成参数，后端真正执行工具，再把结果返回给模型。这样能把模型的语言能力和外部系统连接起来。

### Q4：Tool Calling 容易出什么问题？

回答：

常见问题有：

1. 工具选错，description 写得不清楚时更明显。
2. 参数不合法，比如缺少 required 字段或类型错。
3. 工具结果太长，模型无法有效利用。
4. 工具失败后模型编造结果。
5. 写操作越权，模型误创建、误删除数据。

解决思路是 schema 清晰、参数校验、结果摘要、错误码、工具权限分级和 trace 记录。

### Q5：ReAct 和 Function Calling 的关系？

回答：

ReAct 是一种推理和行动范式，强调 reasoning、action、observation 循环。Function Calling 是实现 action 的一种工程机制。可以用 Function Calling 来落地 ReAct，让模型在推理中调用工具，然后根据工具 observation 继续推理。

### Q6：Agent 为什么需要 memory？

回答：

因为很多任务不是一次性问答。Memory 能保存长期信息，比如用户偏好、项目约束、历史结论、成员风险提醒。没有 memory，模型每次都像第一次见用户，只能依赖当前 prompt。

但 memory 不能无脑存。需要筛选、分类、过期、合并和人工删除，否则会污染上下文。

### Q7：短期记忆和长期记忆区别？

回答：

短期记忆通常是当前会话窗口，用来保持多轮对话连贯；长期记忆是跨会话保留的高价值信息，比如用户偏好、业务事实、项目配置。短期记忆偏原文，长期记忆偏结构化和摘要化。

### Q8：Agent 的可观测性包括什么？

回答：

至少包括：

1. 用户输入和最终输出。
2. 使用的模型和参数。
3. 每一步工具调用的名称、参数、结果、耗时和状态。
4. 错误信息和重试记录。
5. token、延迟、成本等指标。

可观测性是 Agent 从 demo 走向工程系统的重要前提。

### Q9：Agent 如何避免幻觉？

回答：

不能完全避免，只能降低概率和控制风险。常见方法有：

1. 对事实性问题优先检索知识库或数据库。
2. 要求回答带引用。
3. 对工具失败明确告知，不允许编造。
4. 对关键结论做规则校验或人工确认。
5. 建评测集，统计回答准确率和引用正确率。

在 PartyMate 里，我通过 RAG citation、工具结果回填和人工复核来降低风险。

### Q10：Agent 如何做错误恢复？

回答：

可以分层做：

1. 工具参数错误：返回结构化错误，让模型修正参数重试。
2. 网络或模型超时：有限次数重试，超过后降级。
3. 检索为空：换 query、放宽 top_k 或提示用户补充信息。
4. 文件解析失败：进入人工上传或复核流程。
5. 写操作失败：返回失败原因，不让模型假装成功。

## 7. RAG 技术基础高频题

### Q1：RAG 是什么？

回答：

RAG 是 Retrieval-Augmented Generation，检索增强生成。它先从外部知识库检索和问题相关的文档片段，再把这些片段和用户问题一起给大模型生成回答。它适合解决模型知识过时、专业领域知识不足和回答缺少依据的问题。

### Q2：一个完整 RAG 系统有哪些环节？

回答：

离线环节包括：文档收集、清洗、切分、embedding、索引构建。在线环节包括：query 改写或理解、召回、rerank、上下文拼接、LLM 生成、引用回传和评测。

### Q3：向量检索和 BM25 区别？

回答：

向量检索看语义相似度，适合同义表达和自然语言问题；BM25 是稀疏关键词检索，适合精确术语、编号、人名、专有名词。两者互补，所以很多 RAG 系统会做 hybrid search。

### Q4：chunk size 怎么选？

回答：

没有固定答案，要看文档结构和任务。chunk 太小，语义不完整；chunk 太大，召回不精准，还浪费上下文。党务规程这类文档适合按章节、条款、流程节点切，尽量保证每个 chunk 是完整规则单元。最终应该用评测集对比不同 chunk size 的召回率和回答准确率。

### Q5：Rerank 有什么作用？

回答：

初召回通常追求覆盖率，会取比较多候选，比如 top 20。Rerank 再用更精细的模型对 query 和 passage 相关性打分，把真正相关的片段排到前面。这样能提高最终给 LLM 的上下文质量。

### Q6：RAG 可能失败在哪里？

回答：

常见失败点有：

1. 文档清洗失败，噪声进入索引。
2. 分块切断语义。
3. query 和文档表达差距大，召回不到。
4. 召回到了但排序差，相关片段没进上下文。
5. LLM 忽略引用或错误综合。
6. 引用片段本身过时或冲突。

### Q7：怎么评估 RAG？

回答：

可以从检索和生成两层评估。检索层看 recall@k、MRR、nDCG，判断正确片段是否被召回并排在前面。生成层看 answer correctness、faithfulness、citation accuracy，判断回答是否正确、是否忠于引用、引用是否支撑结论。PartyMate 后续可以构造党务规则问答集来做这个评估。

### Q8：RAG 和微调怎么选？

回答：

RAG 适合知识经常变化、需要引用依据、专业文档较多的场景；微调更适合固定风格、固定任务格式或模型能力补齐。党务规程这类知识需要可追溯和可更新，所以优先 RAG，而不是把规程直接微调进模型。

## 8. 后端和 Python 高频题

### Q1：Starlette 请求处理流程？

回答：

Uvicorn 监听端口，收到 HTTP 请求后按 ASGI 协议交给 Starlette app。Starlette 根据 routes 匹配路径，调用对应 async endpoint。endpoint 从 request 里读取 JSON、form、path_params 或 query_params，调用 service 层处理业务，再返回 JSONResponse、FileResponse 或 StreamingResponse。

### Q2：ASGI 是什么？

回答：

ASGI 是 Python 异步 Web 服务接口标准，支持 HTTP、WebSocket、长连接和流式响应。它类似 WSGI 的异步升级版。Starlette、FastAPI 都是 ASGI 框架，Uvicorn 是常用 ASGI server。

### Q3：async / await 有什么用？

回答：

async / await 适合 IO 密集场景，比如 HTTP 请求、数据库访问、文件上传、模型流式响应。它让程序在等待 IO 时释放执行权，提高并发能力。PartyMate 里模型调用、SSE stream、MCP tool call 都适合异步。

### Q4：Python GIL 是什么？

回答：

GIL 是 CPython 的全局解释器锁，它限制同一进程内多个线程同时执行 Python bytecode。对 CPU 密集任务，多线程不一定提升性能；对 IO 密集任务，线程或 async 仍然有用。像 OCR、embedding、rerank 这类 CPU/GPU 重任务，更适合用进程、外部服务或模型本身的并行能力。

### Q5：SQLite 并发有什么限制？

回答：

SQLite 适合本地轻量应用，但写并发能力有限，因为同一时间通常只有一个 writer。PartyMate 是单用户本地应用，所以问题不大。如果变成多用户 Web 服务，需要迁移到 PostgreSQL/MySQL，并设计连接池、事务和并发控制。

### Q6：你怎么分层 PartyMate 代码？

回答：

大致分为：

1. `web/server.py`：API 和静态文件服务，负责请求响应。
2. `services/`：业务流程，比如材料导入、材料核查、会议闭环、成员记忆、Agent trace。
3. `tools/`：Agent 可调用的原子业务工具，比如 doc_check、rag、meeting_summary。
4. `db/repository.py`：SQLite 数据访问。
5. `agent.py`：Agent runtime。

这样路由层不堆业务逻辑，Agent 工具也能复用底层 service。

### Q7：怎么处理文件上传？

回答：

Starlette 里通过 `await request.form()` 读取 multipart form，再从 `UploadFile` 读取内容。保存前检查后缀是否支持，保存到受控上传目录。对 ZIP 解压要检查 resolve 后的路径是否仍在目标目录内，防止路径穿越。

### Q8：HTTP 状态码怎么用？

回答：

常见的：

1. 200：成功。
2. 400：请求参数错误，比如没传 raw text。
3. 404：资源不存在，比如成员或 OCR task 不存在。
4. 500：服务端异常，比如未捕获错误。

PartyMate 里很多 API 会用 JSONResponse 搭配 status_code 返回错误。

### Q3：你说 Agent trace，有什么真实价值？

回答：

Agent 应用如果没有 trace，很难定位问题。用户只看到最后回答，不知道模型是凭空答的，还是查了知识库、调用了材料检查、读取了成员记忆。trace 能帮助开发者排查工具选择、参数、耗时和错误，也能帮助用户回看依据。在党务这种严肃场景里，可追溯性是必要的。

### Q4：你的 RAG 效果怎么证明？

回答：

目前我主要通过功能验证和手工问题测试证明链路可用，比如能召回党务规程片段并返回引用。但如果说严格效果评估，还需要构建标准问答集，评估 recall@k、引用准确率和回答正确率。这是我后续想补的，不会夸大说已经做了完整 benchmark。

### Q5：为什么简历里 PartyMate 写混合检索，但 README 有的地方说关键词？

回答：

项目是迭代开发的，早期 RAG 是规则文本分块和关键词检索，后续我在 `rag.py` 里升级到 ChromaDB + BM25 + rerank 的混合检索。所以文档里如果有不一致，是因为开发文档和 README 更新节奏不完全一致。面试时我会按当前代码实现说明，并承认文档需要同步。

### Q6：你了解 LangGraph 吗？

回答：

了解基本思想。LangGraph 更适合把 Agent 流程建模成状态图，节点可以是 LLM、工具、条件分支或人工确认，边控制状态流转。它比简单 ReAct loop 更适合复杂、可恢复、可审计的工作流。PartyMate 当前还没接 LangGraph，但材料包审核和会议闭环其实很适合后续迁移成图。

### Q7：如果模型一直循环调用工具怎么办？

回答：

工程上要设最大轮次、工具调用次数、超时时间和重复调用检测。PartyMate 当前设置了最多 10 轮。如果模型重复调用同一个工具且参数相同，可以直接终止并提示用户。更进一步可以给每个任务设置状态，避免模型不知道已经完成哪一步。

### Q8：如果 Agent 调错工具怎么办？

回答：

先从 prompt 和 tool schema 优化，比如工具 description 写清适用场景，参数名明确。其次在工具层做参数校验，调错时返回结构化错误。再进一步可以加工具选择评估集，统计不同问题下模型的工具选择准确率。

### Q9：你觉得 Agent 产品最难上线的点是什么？

回答：

我觉得是可靠性和安全性。Demo 里模型能调用工具很酷，但上线时要保证它不会乱写数据、不会编造结果、失败能解释、成本和延迟可控。Agent 需要比普通应用更多的 trace、权限、评测和人工确认机制。

## 12. 技术细节速记

### PartyMate 架构一句话

Python + Starlette 提供本地 Web API，SQLite 存结构化业务状态，Ollama/OpenAI-compatible API 提供模型能力，手写 Agent runtime 负责 ReAct + Tool Calling，多种 service 负责材料、会议、记忆、OCR、trace 等业务闭环。

### PartyMate 请求流

用户在前端发起请求 -> Starlette route 接收 -> 解析 JSON/form/path params -> 调用 service 或 Agent runtime -> service 读写 SQLite/文件/RAG -> 返回 JSON/SSE/file -> 前端渲染。

### Agent 对话流

用户消息 -> SessionMemory 组装历史 -> MemberMemory 组装成员上下文 -> ToolRegistry 提供 schema -> 模型流式返回 -> 如有 tool_calls 则执行工具 -> 记录 trace -> tool result 回填 -> 下一轮模型 -> 最终回答。

### RAG 查询流

用户问题 -> query embedding -> Chroma dense recall -> BM25 sparse recall -> score fusion -> rerank -> top chunks -> 格式化引用 -> 注入 Agent 或返回 API。

### 材料包核查流

ZIP 上传 -> 保存 batch -> 安全解压 -> 支持格式过滤 -> 文件解析/OCR -> 材料类型识别 -> OCR 低置信度进复核 -> 按成员阶段查缺件/重复/阶段冲突/身份冲突 -> 保存检查结果。
